## 设计原则

### 1. 简洁实用

- 只保留必要字段，避免过度设计
- 复杂业务逻辑放在应用层（Java代码）处理

### 2. 适度冗余

- 常用字段适当冗余（如 `view_count`、`post_count`）
- 减少联表查询，提升查询性能

### 3. 统一命名规范

| 规范 | 说明 | 示例 |
|------|------|------|
| 表名 | 小写字母，下划线分隔 | `user`、`product_model` |
| 主键 | 统一使用 `id` | `id BIGINT AUTO_INCREMENT` |
| 外键 | 表名_id | `user_id`、`category_id` |
| 时间字段 | 统一使用 `created_at`、`updated_at` | |
| 逻辑删除 | 统一使用 `deleted` | `deleted TINYINT DEFAULT 0` |
| 状态字段 | 使用 `status` 或 `xxx_status` | `status TINYINT` |

### 4. 状态值约定

| 值 | 含义 |
|------|------|
| 0 | 禁用/未激活/已删除 |
| 1 | 正常/启用/已激活 |
| 2 | 待审核/处理中 |
| 3 | 失败/拒绝 |

## 用户模块

用户模块包含 3 张表，负责用户基础信息、扩展资料和收货地址管理。

### 表清单

| 表名 | 说明 | 关系 |
|------|------|------|
| `user` | 用户基础信息 | 主表 |
| `user_profile` | 用户扩展资料 | 属于 user（1:1） |
| `user_address` | 收货地址 | 属于 user（1:N） |

---

### 1. user（用户表）

**用途**：存储用户的基础信息，如账号、昵称、头像等。

```
┌─────────────────────────────────────────────────────────────┐
│                        user 表                               │
├─────────────────────────────────────────────────────────────┤
│  id          │ 用户ID，自增主键                               │
│  phone       │ 手机号，唯一，用于登录                          │
│  password    │ 密码，加密存储（BCrypt）                        │
│  nickname    │ 昵称                                          │
│  avatar      │ 头像URL                                       │
│  gender      │ 性别：0-未知 1-男 2-女                         │
│  birthday    │ 生日                                          │
│  signature   │ 个性签名                                       │
│  status      │ 状态：1-正常 0-禁用 2-待激活                    │
│  last_login_time  │ 最后登录时间                          │
│  last_login_ip    │ 最后登录IP                             │
│  created_at  │ 创建时间                                       │
│  updated_at  │ 更新时间                                       │
│  deleted     │ 逻辑删除：0-未删除 1-已删除                     │
└─────────────────────────────────────────────────────────────┘
```

**字段详解**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `id` | BIGINT | 是 | 主键，自增 |
| `phone` | VARCHAR(20) | 否 | 手机号，唯一索引，允许为空（第三方登录场景） |
| `password` | VARCHAR(255) | 否 | 加密后的密码，使用 BCrypt 算法 |
| `nickname` | VARCHAR(50) | 否 | 昵称，可重复 |
| `avatar` | VARCHAR(255) | 否 | 头像图片地址 |
| `gender` | TINYINT | 否 | 0-未知 1-男 2-女，默认0 |
| `birthday` | DATE | 否 | 出生日期 |
| `signature` | VARCHAR(255) | 否 | 个性签名 |
| `status` | TINYINT | 是 | 1-正常 0-禁用 2-待激活，默认1 |
| `deleted` | TINYINT | 是 | 逻辑删除标记，默认0 |

**常见查询场景**：

```sql
-- 根据手机号查询用户
SELECT * FROM user WHERE phone = '13800138000' AND deleted = 0;

-- 更新最后登录信息
UPDATE user SET last_login_time = NOW(), last_login_ip = '127.0.0.1' 
WHERE id = 1;
```

### 2. user_profile（用户扩展资料表）

**用途**：存储用户的实名认证、店铺认证等扩展信息。

```
┌─────────────────────────────────────────────────────────────┐
│                     user_profile 表                          │
├─────────────────────────────────────────────────────────────┤
│  id                 │ 主键                                   │
│  user_id            │ 用户ID，唯一，关联 user 表              │
│  real_name          │ 真实姓名                               │
│  id_card            │ 身份证号                               │
│  id_card_front      │ 身份证正面照URL                        │
│  id_card_back       │ 身份证背面照URL                        │
│  real_name_status   │ 实名状态：0-未认证 1-认证中 2-已认证 3-失败 │
│  real_name_time     │ 认证时间                               │
│  real_name_reason   │ 认证失败原因                           │
│  shop_auth_status   │ 店铺认证状态                           │
│  shop_auth_time     │ 店铺认证时间                           │
│  shop_auth_reason   │ 店铺认证失败原因                       │
│  province           │ 省份                                   │
│  city               │ 城市                                   │
│  district           │ 区县                                   │
│  created_at         │ 创建时间                               │
│  updated_at         │ 更新时间                               │
└─────────────────────────────────────────────────────────────┘
```

**字段详解**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | BIGINT | 关联 user.id，一对一关系 |
| `real_name` | VARCHAR(50) | 真实姓名，实名认证通过后填写 |
| `id_card` | VARCHAR(20) | 身份证号，需脱敏显示 |
| `real_name_status` | TINYINT | 0-未认证 1-认证中 2-已认证 3-认证失败 |
| `shop_auth_status` | TINYINT | 开店需要单独认证 |

**认证流程说明**：

```
用户提交实名信息 → real_name_status = 1（认证中）
        ↓
    后台审核
        ↓
   ┌────┴────┐
   ↓         ↓
 通过       拒绝
   ↓         ↓
status=2   status=3
           reason=原因
```

### 3. user_address（用户收货地址表）

**用途**：存储用户的收货地址，支持多个地址和默认地址。

```
┌─────────────────────────────────────────────────────────────┐
│                     user_address 表                          │
├─────────────────────────────────────────────────────────────┤
│  id          │ 地址ID，主键                                  │
│  user_id     │ 用户ID，关联 user 表                          │
│  name        │ 收货人姓名                                    │
│  phone       │ 收货人电话                                    │
│  province    │ 省份                                         │
│  city        │ 城市                                         │
│  district    │ 区/县                                        │
│  detail      │ 详细地址                                      │
│  is_default  │ 是否默认：0-否 1-是                           │
│  created_at  │ 创建时间                                      │
│  updated_at  │ 更新时间                                      │
│  deleted     │ 逻辑删除                                      │
└─────────────────────────────────────────────────────────────┘
```

**业务规则**：

1. 一个用户可以有多个地址
2. 只能有一个默认地址
3. 设置新默认地址时，需先取消旧的默认地址

**设置默认地址示例**：

```sql
-- 步骤1：取消该用户所有默认地址
UPDATE user_address SET is_default = 0 WHERE user_id = 1;

-- 步骤2：设置新的默认地址
UPDATE user_address SET is_default = 1 WHERE id = 5 AND user_id = 1;
```

## 产品库模块

产品库模块参考 ZOL 中关村在线的设计思路，实现多级分类和动态参数。

### 设计思路

```
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│   分类体系    │ ──→  │   属性模板    │ ──→  │   产品型号    │
│  (多级树形)   │      │  (动态参数)   │      │  (具体产品)   │
└──────────────┘      └──────────────┘      └──────────────┘
```

**核心概念**：

1. **分类**：手机 → 游戏手机 → 电竞旗舰（三级分类）
2. **属性模板**：屏幕尺寸、处理器、电池容量...（不同分类有不同属性）
3. **产品型号**：iPhone 15 Pro Max、小米14 Ultra...（具体产品）
4. **属性值**：iPhone 15 Pro Max 的屏幕尺寸是 6.7 英寸

---

### 表清单

| 表名 | 说明 |
|------|------|
| `category` | 设备分类（支持多级） |
| `brand` | 品牌 |
| `brand_category` | 品牌分类关联（一个品牌可属于多个分类） |
| `attribute` | 属性模板（定义属性名称和类型） |
| `attribute_option` | 属性选项值（单选/多选类型的选项） |
| `category_attribute` | 分类属性关联（哪些分类有哪些属性） |
| `product_model` | 产品型号 |
| `product_model_attribute` | 产品型号属性值 |

---

### 1. category（设备分类表）

**用途**：存储设备的多级分类，如"手机 → 游戏手机 → 电竞旗舰"。

```
┌─────────────────────────────────────────────────────────────┐
│                      category 表                             │
├─────────────────────────────────────────────────────────────┤
│  id         │ 分类ID，主键                                   │
│  name       │ 分类名称                                       │
│  parent_id  │ 父分类ID，0表示顶级分类                         │
│  level      │ 层级：1-一级 2-二级 3-三级                     │
│  icon       │ 分类图标URL                                    │
│  image      │ 分类图片URL                                    │
│  sort       │ 排序，数字越小越靠前                           │
│  status     │ 状态：1-启用 0-禁用                            │
│  created_at │ 创建时间                                       │
│  updated_at │ 更新时间                                       │
└─────────────────────────────────────────────────────────────┘
```

**分类层级示例**：

```
手机（id=1, parent_id=0, level=1）
├── 游戏手机（id=10, parent_id=1, level=2）
│   ├── 电竞旗舰（id=100, parent_id=10, level=3）
│   └── 游戏水桶（id=101, parent_id=10, level=3）
├── 轻薄手机（id=11, parent_id=1, level=2）
│   └── 超薄机身（id=102, parent_id=11, level=3）
└── 拍照手机（id=12, parent_id=1, level=2）
    ├── 影像旗舰（id=103, parent_id=12, level=3）
    └── 人像手机（id=104, parent_id=12, level=3）

电脑（id=2, parent_id=0, level=1）
├── 游戏本（id=20, parent_id=2, level=2）
├── 轻薄本（id=21, parent_id=2, level=2）
└── 台式机（id=22, parent_id=2, level=2）
```

**查询某分类的所有子分类**：

```sql
-- 查询"手机"下的所有二级分类
SELECT * FROM category WHERE parent_id = 1 AND status = 1;

-- 查询"游戏手机"下的所有三级分类
SELECT * FROM category WHERE parent_id = 10 AND status = 1;

-- 查询某分类的完整路径（从根到叶）
-- 需要在应用层递归查询，或使用 MySQL 8.0 的 WITH RECURSIVE
```

### 6. category_attribute（分类属性关联表）

**用途**：定义哪些分类需要填写哪些属性，这是"属性模板"的核心。

```
┌─────────────────────────────────────────────────────────────┐
│                 category_attribute 表                        │
├─────────────────────────────────────────────────────────────┤
│  id           │ 主键                                        │
│  category_id  │ 分类ID                                      │
│  attribute_id │ 属性ID                                      │
│  is_required  │ 是否必填：0-否 1-是                          │
│  is_filter    │ 是否作为筛选条件：0-否 1-是                   │
│  is_show      │ 是否在详情页显示：0-否 1-是                   │
│  sort         │ 排序                                        │
└─────────────────────────────────────────────────────────────┘
```

**示例**：

```
"游戏手机"分类的属性模板：
├── 屏幕尺寸（必填、筛选、显示）
├── 处理器型号（必填、筛选、显示）
├── 屏幕类型（选填、筛选、显示）
├── 电池容量（必填、筛选、显示）
├── 是否支持快充（选填、不筛选、显示）
└── 网络制式（选填、筛选、显示）

"耳机"分类的属性模板：
├── 佩戴方式（必填、筛选、显示）
├── 连接方式（必填、筛选、显示）
├── 续航时间（选填、筛选、显示）
└── 是否降噪（选填、筛选、显示）
```

**查询某分类的属性模板**：

```sql
SELECT a.*, ca.is_required, ca.is_filter, ca.is_show
FROM attribute a
JOIN category_attribute ca ON a.id = ca.attribute_id
WHERE ca.category_id = 10  -- 游戏手机
ORDER BY ca.sort;
```

**冗余字段说明**：

```

├─────────────────────────────────────────────────────────────┤
```

**value 字段存储规则**：

| input_type | value 存储方式 | 示例 |

### 1. conversation（会话表）

**用途**：存储两个用户之间的会话信息。

```
┌─────────────────────────────────────────────────────────────┐
│                   conversation 表                            │
├─────────────────────────────────────────────────────────────┤
│  id               │ 会话ID，主键                             │
│  user_id          │ 用户ID（当前用户）                       │
│  target_id        │ 对方用户ID                               │
│  last_message     │ 最后一条消息内容                         │
│  last_message_type│ 最后消息类型 1-文本 2-图片 3-商品 4-订单 │
│  last_message_time│ 最后消息时间                             │
│  unread_count     │ 未读消息数                               │
│  is_pinned        │ 是否置顶 0-否 1-是                       │
│  is_muted         │ 是否免打扰 0-否 1-是                     │

└─────────────────────────────────────────────────────────────┘
```

**设计说明**：

```
用户A 和 用户B 聊天，会产生两条会话记录：

conversation 1: user_id=A, target_id=B  （A看到的会话列表）
conversation 2: user_id=B, target_id=A  （B看到的会话列表）

这样设计的好处：
- 各自管理自己的会话（删除、置顶、免打扰互不影响）
- 各自有独立的未读数
```

**查询会话列表**：

```sql
SELECT c.*, u.nickname, u.avatar 
FROM conversation c
JOIN user u ON c.target_id = u.id
WHERE c.user_id = 1 AND c.is_deleted = 0
ORDER BY c.is_pinned DESC, c.last_message_time DESC;
```

```
│    (Post)    │      │  (Comment)   │      │(CommentLike) │
└──────────────┘      └──────────────┘      └──────────────┘
       │
       │
       ▼
┌──────────────┐
│   帖子点赞   │
│  (PostLike)  │
└──────────────┘
```

**核心概念**：

1. **社区**：用户创建的主题社区，如"iPhone交流群"
2. **帖子**：社区内发布的内容
3. **评论**：对帖子的评论，支持嵌套回复
4. **点赞**：对帖子或评论的点赞

## 附录：常见问题

### Q1: 为什么属性值要分开存储，而不是用 JSON 字段？

**答**：
- 优点：支持按属性筛选、排序
- 缺点：查询需要联表
- 如果不需要按属性筛选，可以用 JSON 字段简化设计

### Q2: 分类层级有限制吗？

**答**：
- 当前设计支持无限层级
- 但建议最多 3 级，层级太深用户体验不好
- 通过 `level` 字段可以限制层级深度

### Q3: 冗余字段会不会导致数据不一致？

**答**：
- 需要在应用层维护一致性
- 例如：用户关注产品时，同时更新 `follow_count`
- 可以使用数据库事务保证原子性

### Q4: 为什么会话表要存两份记录？

**答**：
- 用户A和用户B聊天，各自有一条会话记录
- 好处：各自的未读数、置顶、免打扰、删除操作互不影响
- 缺点：发送消息时需要更新两条会话记录

### Q5: 商品快照有什么用？

**答**：
- 下单时保存商品信息快照
- 避免商品修改/删除后影响历史订单
- 便于订单详情展示和纠纷处理
